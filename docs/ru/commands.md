# Команды

Это справочник по слэш-командам OpenSpec. Эти команды вызываются в интерфейсе чата вашего ИИ-ассистента по программированию (например, Claude Code, Cursor, Windsurf).

Сведения о паттернах рабочих процессов и о том, когда использовать каждую команду, см. в разделе [Рабочие процессы](workflows.md). Сведения о командах CLI см. в разделе [CLI](cli.md).

## Быстрый справочник

### Стандартный быстрый путь (профиль `core`)

| Команда | Назначение |
|---------|------------|
| `/opsx:propose` | Создать изменение и сгенерировать артефакты планирования за один шаг |
| `/opsx:explore` | Продумать идеи перед тем, как приступить к внесению изменения |
| `/opsx:apply` | Выполнить задачи из изменения |
| `/opsx:update` | Пересмотреть артефакты планирования изменения и поддерживать их согласованность |
| `/opsx:sync` | Объединить дельта-спецификации в основные спецификации |
| `/opsx:archive` | Архивировать завершенное изменение |

### Расширенные команды рабочих процессов (выбор пользовательского рабочего процесса)

| Команда | Назначение |
|---------|------------|
| `/opsx:new` | Создать каркас нового изменения |
| `/opsx:continue` | Создать следующий артефакт на основе зависимостей |
| `/opsx:ff` | Быстрый переход: создать все артефакты планирования сразу |
| `/opsx:verify` | Проверить, что реализация соответствует артефактам |
| `/opsx:bulk-archive` | Архивировать несколько изменений сразу |
| `/opsx:onboard` | Обучающее руководство по полному рабочему процессу |

Стандартным глобальным профилем является `core`. Чтобы включить расширенные команды рабочих процессов, выполните команду `openspec config profile`, выберите нужные рабочие процессы, а затем выполните `openspec update` в вашем проекте.

## Справочник команд

### `/opsx:propose`

Создайте новое изменение и сгенерируйте артефакты планирования за один шаг. Это команда запуска по умолчанию в профиле `core`.

**Синтаксис:**
```text
/opsx:propose [change-name-or-description]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name-or-description` | Нет | Имя в kebab-case или описание изменения на обычном языке |

**Что она делает:**
- Создает директорию `openspec/changes/<change-name>/`
- Генерирует артефакты, необходимые перед реализацией (для `spec-driven`: proposal, specs, design, tasks)
- Останавливается, когда изменение готово к запуску `/opsx:apply`

**Пример:**
```text
Вы: /opsx:propose add-dark-mode

ИИ:  Создана openspec/changes/add-dark-mode/
     ✓ proposal.md
     ✓ specs/ui/spec.md
     ✓ design.md
     ✓ tasks.md
     Ready for implementation. Run /opsx:apply.
```

**Советы:**
- Используйте эту команду для самого быстрого сквозного пути
- Если вам нужен пошаговый контроль над артефактами, включите расширенные рабочие процессы и используйте `/opsx:new` + `/opsx:continue`

### `/opsx:explore`

> **Начните отсюда, если вы не уверены.** Explore — это партнер для размышлений без риска: он читает вашу кодовую базу, сравнивает варианты и превращает размытое представление о идее в конкретный план до создания любого изменения. Входит в стандартный профиль. Полный обзор сценария и дополнительные примеры см. в руководстве [Сначала исследуйте](explore.md).

Обдумывайте идеи, исследуйте проблемы и уточняйте требования перед тем, как приступить к изменению.

**Синтаксис:**
```
/opsx:explore [topic]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `topic` | Нет | Тема, которую вы хотите изучить или исследовать |

**Что она делает:**
- Открывает исследовательский диалог без требований к структуре
- Исследует кодовую базу для ответа на вопросы
- Сравнивает варианты и подходы
- Создает визуальные диаграммы для прояснения мыслительного процесса
- Может перейти к `/opsx:propose` (по умолчанию) или `/opsx:new` (расширенный рабочий процесс), когда идеи сформируются

**Пример:**
```text
Вы: /opsx:explore

ИИ:  Что вы хотите исследовать?

Вы: Как нам организовать аутентификацию для мобильного приложения?

ИИ:  Позвольте мне изучить вашу текущую конфигурацию аутентификации...
     [Анализирует кодовую базу]

     Я вижу, что для веб-приложения вы используете аутентификацию на основе сессий.
     Для мобильного приложения у нас есть три основных варианта:

     1. JWT-токены (без состояния, работают в офлайн-режиме)
     2. OAuth2 (если вам нужен вход через сторонние сервисы)
     3. Расширить текущие сессии (проще в реализации, но требует постоянного подключения)

     В вашем API уже настроен CORS. Какой вариант вас интересует?

Вы: Давайте остановимся на JWT. Можно ли начать для этого изменение?

ИИ:  Я готов, когда вы будете. Запустите /opsx:propose add-jwt-auth, чтобы начать.
```

**Советы:**
- Используйте, когда требования неясны или вам нужно провести исследование
- Во время исследования артефакты не создаются
- Подходит для сравнения нескольких подходов перед принятием решения
- Может читать файлы и искать по кодовой базе

### `/opsx:new`

Создайте каркас нового изменения. Создает папку изменения и ожидает, пока вы сгенерируете артефакты с помощью `/opsx:continue` или `/opsx:ff`.

Эта команда входит в набор расширенных рабочих процессов (не включена в стандартный профиль `core`).

**Синтаксис:**
```
/opsx:new [change-name] [--schema <schema-name>]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Имя для папки изменения (запрашивается, если не указано) |
| `--schema` | Нет | Схема рабочего процесса для использования (по умолчанию: из конфигурации или `spec-driven`) |

**Что она делает:**
- Создает директорию `openspec/changes/<change-name>/`
- Создает файл метаданных `.openspec.yaml` в папке изменения
- Отображает первый шаблон артефакта, готовый к созданию
- Запрашивает имя изменения и схему, если они не указаны

**Что создается:**
```
openspec/changes/<change-name>/
└── .openspec.yaml    # Метаданные изменения (схема, дата создания)
```

**Пример:**
```
Вы: /opsx:new add-dark-mode

ИИ:  Создана openspec/changes/add-dark-mode/
     Схема: spec-driven

     Готово к созданию: proposal
     Используйте /opsx:continue для создания, или /opsx:ff для создания всех артефактов.
```

**Советы:**
- Используйте описательные имена: `add-feature`, `fix-bug`, `refactor-module`
- Избегайте общих имен, например `update`, `changes`, `wip`
- Схему также можно указать в конфигурации проекта (`openspec/config.yaml`)

### `/opsx:continue`

Создайте следующий артефакт в цепочке зависимостей. Создает по одному артефакту за раз для постепенного прогресса.

**Синтаксис:**
```
/opsx:continue [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение продолжить (определяется из контекста, если не указано) |

**Что она делает:**
- Запрашивает граф зависимостей артефактов
- Отображает, какие артефакты готовы, а какие заблокированы
- Создает первый готовый артефакт
- Читает файлы зависимостей для получения контекста
- Отображает, что станет доступно после создания

**Пример:**
```
Вы: /opsx:continue

ИИ:  Изменение: add-dark-mode

     Статус артефактов:
     ✓ proposal    (готово)
     ◆ specs       (готово)
     ◆ design      (готово)
     ○ tasks       (заблокировано — требуется: specs)

     Создание specs...

     [Читает proposal.md, создает specs на основе требований]

     ✓ Создан openspec/changes/add-dark-mode/specs/ui/spec.md

     Теперь доступно: tasks
     Запустите /opsx:continue для создания следующего артефакта.
```

**Советы:**
- Используйте, если хотите проверить каждый артефакт перед переходом к следующему
- Подходит для сложных изменений, когда вам нужен полный контроль
- Несколько артефактов могут стать готовыми одновременно
- Вы можете редактировать созданные артефакты перед продолжением

---

### `/opsx:ff`

Пропустите создание артефактов вперед. Создает все артефакты планирования сразу.

**Синтаксис:**
```
/opsx:ff [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение пропустить вперед (определяется из контекста, если не указано) |

**Что она делает:**
- Создает все артефакты в порядке зависимостей
- Отслеживает прогресс с помощью списка задач
- Останавливается, когда все артефакты, требующие применения, готовы
- Читает каждую зависимость перед созданием следующего артефакта

**Пример:**
```
Вы: /opsx:ff add-dark-mode

ИИ:  Пропуск создания артефактов для add-dark-mode...

     ✓ Создание proposal.md
     ✓ Создание specs/ui/spec.md
     ✓ Создание design.md
     ✓ Создание tasks.md

     Все артефакты планирования готовы!
     Готово к реализации. Запустите /opsx:apply, чтобы начать.
```

**Советы:**
- Используйте, если у вас есть четкое представление о том, что вы создаете
- Быстрее, чем `/opsx:continue`, для простых изменений
- Вы все еще можете редактировать артефакты после создания
- Подходит для небольших и средних функций

---

### `/opsx:apply`

Реализуйте задачи из изменения. Последовательно выполняет задачи из списка, пишет код и отмечает выполненные пункты.

**Синтаксис:**
```
/opsx:apply [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение реализовать (определяется из контекста, если не указано) |

**Что она делает:**
- Читает `tasks.md` и определяет невыполненные задачи
- Последовательно выполняет задачи одну за другой
- Пишет код, создает файлы, запускает тесты при необходимости
- Отмечает выполненные задачи галочками `[x]`

**Пример:**
```
Вы: /opsx:apply

ИИ:  Реализация add-dark-mode...

     Чтение tasks.md:
     - [ ] 1.1 Создать ThemeContext
     - [ ] 1.2 Добавить пользовательские свойства CSS
     - [ ] 1.3 Реализовать сохранение в localStorage

     Выполнение задачи 1.1: Создать ThemeContext...
     [Создает src/context/ThemeContext.tsx]

     ✓ 1.1 Готово

     Выполнение задачи 1.2: Добавить пользовательские свойства CSS...
     [Обновляет src/styles/globals.css]

     ✓ 1.2 Готово
     ...
```

**Советы:**
- Может возобновить работу с того места, где вы остановились, если процесс был прерван
- Используйте для параллельных изменений, указав имя изменения
- Состояние выполнения отслеживается галочками в `tasks.md`

---

### `/opsx:update`

Отредактируйте существующие артефакты планирования изменения и поддерживайте их согласованность между собой. Только артефакты планирования — команда никогда не редактирует код.

**Синтаксис:**
```text
/opsx:update [change-name]
```

**Аргументы:**

| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение обновить (определяется из контекста, если не указано) |

**Что она делает:**

- Читает артефакты изменения с помощью команды `openspec status --change <name> --json`
- Применяет запрошенное вами изменение или проверяет артефакты на противоречия, если вы не указали конкретное изменение
- Согласовывает остальные существующие артефакты в любом направлении (редактирование дизайна может повлиять на proposal)
- Подтверждает каждое редактирование с вами перед записью, по одному артефакту за раз
- Завершается рекомендацией следующего шага: `/opsx:continue` (отсутствуют артефакты), `/opsx:apply` (переход к реализации измененного плана) или `/opsx:archive` (все готово)

**Пример:**

```text
Вы: /opsx:update add-dark-mode - теперь мы храним тему в cookie, а не в localStorage

ИИ:  Чтение артефактов add-dark-mode...

     В design.md дважды упоминается localStorage; задача 1.3 в tasks.md
     касается сохранения в localStorage; в proposal нет упоминания хранилища.

     Предлагаемые изменения:
     1. design.md — заменить решение о localStorage на хранение в cookie
     2. tasks.md — переформулировать задачу 1.3 на сохранение в cookie

     Применить изменение 1? (design.md)
```

**Советы:**

- Она не создает отсутствующие артефакты — для этого используйте `/opsx:continue`
- Если изменение уже было реализовано, дополнительно запустите `/opsx:apply`, чтобы код соответствовал измененному плану
- Если ваше редактирование меняет *намерение* изменения, начните с нового изменения вместо этого (см. раздел [Когда обновлять, а когда начинать заново](opsx.md#when-to-update-vs-start-fresh))

### `/opsx:verify`

Проверьте, что реализация соответствует артефактам вашего изменения. Проверяет полноту, корректность и согласованность.

**Синтаксис:**
```
/opsx:verify [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение проверить (определяется из контекста, если не указано) |

**Что она делает:**
- Проверяет три аспекта качества реализации
- Ищет в кодовой базе подтверждения реализации
- Сообщает о проблемах, разделенных на категории КРИТИЧЕСКИЕ, ПРЕДУПРЕЖДЕНИЯ и РЕКОМЕНДАЦИИ
- Не блокирует архивацию, но выявляет проблемы

**Аспекты проверки:**

| Аспект | Что проверяется |
|-----------|-------------------|
| **ПОЛНОТА** | Все задачи выполнены, все требования реализованы, все сценарии охвачены |
| **КОРРЕКТНОСТЬ** | Реализация соответствует назначению спецификаций, обработаны крайние случаи |
| **СОГЛАСОВАННОСТЬ** | Решения по дизайну отражены в коде, шаблоны проектирования согласованы |

**Пример:**
```
Вы: /opsx:verify

ИИ:  Проверка add-dark-mode...

     ПОЛНОТА
     ✓ Все 8 задач в tasks.md отмечены как выполненные
     ✓ Все требования в specs имеют соответствующий код
     ⚠ У сценария «Обнаружение системных предпочтений» нет покрытия тестами

     КОРРЕКТНОСТЬ
     ✓ Реализация соответствует назначению спецификаций
     ✓ Крайние случаи из сценариев обработаны
     ✓ Состояния ошибок соответствуют определениям в спецификациях

     СОГЛАСОВАННОСТЬ
     ✓ Решения по дизайну отражены в структуре кода
     ✓ Соглашения об именовании соответствуют design.md
     ⚠ В дизайне упоминаются «CSS-переменные», но в реализации используются классы Tailwind

     СВОДКА
     ─────────────────────────────
     Критические проблемы: 0
     Предупреждения: 2
     Готово к архивации: Да (с предупреждениями)

     Рекомендации:
     1. Добавить тест для обнаружения системных предпочтений
     2. Обновить design.md, чтобы он отражал использование Tailwind, или переработать код для использования CSS-переменных
```

**Советы:**
- Запускайте перед архивацией, чтобы заранее выявить несоответствия
- Предупреждения не блокируют архивацию, но указывают на потенциальные проблемы
- Подходит для проверки работы ИИ перед фиксацией изменений
- Может выявить расхождения между артефактами и реализацией

---

### `/opsx:sync`

**Необязательная команда.** Объедините дельта-спецификации из изменения в основные спецификации. Архивация предложит выполнить синхронизацию при необходимости, поэтому обычно запускать эту команду вручную не требуется.

**Синтаксис:**
```
/opsx:sync [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение синхронизировать (определяется из контекста, если не указано) |

**Что она делает:**
- Читает дельта-спецификации из папки изменения
- Разбирает секции ADDED/MODIFIED/REMOVED/RENAMED
- Объединяет изменения в основную директорию `openspec/specs/`
- Сохраняет существующее содержимое, не упомянутое в дельта-спецификации
- Не архивирует изменение (остается активным)

**Пример:**
```text
Вы: /opsx:sync

ИИ:  Синхронизация дельта-спецификаций add-dark-mode...

     Чтение: openspec/changes/add-dark-mode/specs/ui/spec.md
     Цель:  openspec/specs/ui/spec.md

     Изменения для применения:
     ✓ ADDED: Требование «Выбор темы» (2 сценария)
     ✓ ADDED: Требование «Доступность» (1 сценарий)

     Объединение...
     ✓ openspec/specs/ui/spec.md обновлен

     Изменение остается активным. Запустите /opsx:archive, когда будете готовы завершить.
```

**Когда использовать вручную:**

| Сценарий | Использовать синхронизацию? |
|----------|-----------|
| Длительное изменение, нужно иметь спецификации в основной ветке перед архивацией | Да |
| Несколько параллельных изменений требуют обновленных базовых спецификаций | Да |
| Хотите предварительно просмотреть/проверить объединение отдельно | Да |
| Быстрое изменение, сразу переходите к архивации | Нет (архивация обрабатывает это автоматически) |

**Советы:**
- Синхронизация интеллектуальная, а не простое копирование-вставка
- Может добавлять сценарии к существующим требованиям без дублирования
- Изменение остается активным после синхронизации (не архивируется)
- Большинству пользователей никогда не придется вызывать эту команду напрямую — архивация предложит выполнить синхронизацию при необходимости

---

### `/opsx:archive`

Архивируйте завершенное изменение. Завершает изменение и перемещает его в папку архива.

**Синтаксис:**
```
/opsx:archive [change-name]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-name` | Нет | Какое изменение архивировать (определяется из контекста, если не указано) |

**Что она делает:**
- Проверяет статус завершения артефактов
- Проверяет завершение задач (выдает предупреждение, если есть невыполненные)
- Предлагает синхронизировать дельта-спецификации, если они еще не синхронизированы
- Перемещает папку изменения в `openspec/changes/archive/YYYY-MM-DD-<name>/`
- Сохраняет все артефакты для аудита

**Пример:**
```text
Вы: /opsx:archive

ИИ:  Архивация add-dark-mode...

     Статус артефактов:
     ✓ proposal.md существует
     ✓ specs/ существует
     ✓ design.md существует
     ✓ tasks.md существует (8/8 задач выполнено)

     Дельта-спецификации: Еще не синхронизированы
     → Синхронизировать сейчас? (рекомендуется)

Вы: Да

ИИ:  ✓ Спецификации синхронизированы в openspec/specs/ui/spec.md
     ✓ Перемещено в openspec/changes/archive/2025-01-24-add-dark-mode/

     Изменение успешно архивировано.
```

**Советы:**
- Архивация не блокируется из-за невыполненных задач, но выдает предупреждение
- Дельта-спецификации можно синхронизировать во время архивации или заранее
- Архивированные изменения сохраняются для истории
- Сначала используйте `/opsx:verify`, чтобы выявить проблемы

---

### `/opsx:bulk-archive`

Архивируйте несколько завершенных изменений сразу. Обрабатывает конфликты спецификаций между изменениями.

**Синтаксис:**
```
/opsx:bulk-archive [change-names...]
```

**Аргументы:**
| Аргумент | Обязательный | Описание |
|----------|--------------|----------|
| `change-names` | Нет | Конкретные изменения для архивации (запрашивает выбор, если не указаны) |

**Что она делает:**
- Выводит список всех завершенных изменений
- Проверяет каждое изменение перед архивацией
- Обнаруживает конфликты спецификаций между изменениями
- Разрешает конфликты, проверяя, что фактически реализовано
- Архивирует в хронологическом порядке

**Пример:**
```
Вы: /opsx:bulk-archive

ИИ:  Найдено 3 завершенных изменения:
     - add-dark-mode (8/8 задач выполнено)
     - fix-login-bug (3/3 задач выполнено)
     - update-footer (2/2 задач выполнено)

     Проверка на конфликты спецификаций...
     ⚠ add-dark-mode и update-footer затрагивают specs/ui/

     Анализ кодовой базы для разрешения конфликтов...
     Оба изменения реализованы. Объединение будет выполнено в хронологическом порядке:
     1. add-dark-mode (создано 20 янв.)
     2. update-footer (создано 22 янв.)

     Архивировать все 3 изменения?

Вы: Да

ИИ:  ✓ Архивировано add-dark-mode
     ✓ Архивировано fix-login-bug
     ✓ Архивировано update-footer
     Спецификации объединены: add-dark-mode → update-footer
```

**Советы:**
- Подходит для параллельных рабочих потоков
- Разрешение конфликтов выполняется автоматически (проверяет кодовую базу)
- Изменения архивируются в порядке создания
- Запрашивает подтверждение перед перезаписью содержимого спецификаций

### `/opsx:onboard`

Сопровождаемое ознакомление с полным рабочим процессом OpenSpec. Интерактивный учебник с использованием вашей реальной кодовой базы.

**Синтаксис:**
```
/opsx:onboard
```

**Что она делает:**
- Последовательно проходит полный цикл рабочего процесса с комментариями
- Сканирует вашу кодовую базу в поисках реальных возможностей для улучшения
- Создает реальное изменение с настоящими артефактами
- Реализует реальную работу (небольшие, безопасные изменения)
- Архивирует завершенное изменение
- Объясняет каждый шаг по мере его выполнения

**Этапы:**
1. Приветствие и анализ кодовой базы
2. Поиск возможности для улучшения
3. Создание изменения (`/opsx:new`)
4. Написание proposal
5. Создание specs
6. Написание design
7. Создание tasks
8. Реализация задач (`/opsx:apply`)
9. Проверка реализации
10. Архивирование изменения
11. Сводка и следующие шаги

**Пример:**
```
Вы: /opsx:onboard

ИИ:  Добро пожаловать в OpenSpec!

     Я проведу вас через полный рабочий процесс, используя вашу реальную кодовую базу.
     Мы найдем что-то небольшое для улучшения, создадим для этого соответствующее изменение,
     реализуем его и заархивируем.

     Позвольте мне просканировать вашу кодовую базу в поисках возможностей...

     [Анализирует кодовую базу]

     Я нашел несколько вещей, над которыми можно поработать:
     1. Добавить валидацию входных данных в контактную форму
     2. Улучшить сообщения об ошибках в потоке аутентификации
     3. Добавить состояния загрузки для асинхронных кнопок

     Что вас интересует? (или предложите что-то другое)
```

**Советы:**
- Идеально подходит для новых пользователей, изучающих рабочий процесс
- Использует реальный код, а не учебные примеры
- Создает реальное изменение, которое вы можете сохранить или удалить
- Занимает 15–30 минут на выполнение

## Синтаксис команд в зависимости от ИИ-инструмента

Различные ИИ-инструменты используют немного разный синтаксис команд. Используйте формат, соответствующий вашему инструменту:

| Инструмент | Пример синтаксиса |
|------------|-------------------|
| Claude Code | `/opsx:propose`, `/opsx:apply` |
| Cursor | `/opsx-propose`, `/opsx-apply` |
| Windsurf | `/opsx-propose`, `/opsx-apply` |
| Copilot (IDE) | `/opsx-propose`, `/opsx-apply` |
| CodeArts | Вызовы на основе навыков, например `/openspec-propose`, `/openspec-apply-change` (не генерируются файлы команд `opsx-*`) |
| Codex | Вызовы на основе навыков из `.codex/skills/openspec-*` (не генерируются файлы подсказок `opsx-*`) |
| Oh My Pi | `/opsx-propose`, `/opsx-apply` |
| Kimi Code | Вызовы на основе навыков, например `/skill:openspec-propose`, `/skill:openspec-apply-change` (не генерируются файлы команд `opsx-*`) |
| Trae | `/opsx-propose`, `/opsx-apply` |

Намерение одинаково для всех инструментов, но способ представления команд может отличаться в зависимости от интеграции.

> **Примечание:** Команды GitHub Copilot (`.github/prompts/*.prompt.md`) доступны только в расширениях для IDE (VS Code, JetBrains, Visual Studio). GitHub Copilot CLI в настоящее время не поддерживает пользовательские файлы подсказок — подробности и обходные пути см. в разделе [Поддерживаемые инструменты](supported-tools.md).

## Устаревшие команды

Эти команды используют старый рабочий процесс «все сразу». Они по-прежнему работают, но рекомендуется использовать команды OPSX.

| Команда | Назначение |
|---------|------------|
| `/openspec:proposal` | Создать все артефакты сразу (предложение, спецификации, дизайн, задачи) |
| `/openspec:apply` | Реализовать изменение |
| `/openspec:archive` | Архивировать изменение |

**Когда использовать устаревшие команды:**
- Существующие проекты, использующие старый рабочий процесс
- Простые изменения, для которых не требуется поэтапное создание артефактов
- Предпочтение подходу «всё или ничего»

**Миграция на OPSX:**
Работу над устаревшими изменениями можно продолжить с помощью команд OPSX. Структура артефактов совместима.

## Устранение неполадок

### «Изменение не найдено»

Команда не смогла определить, с каким изменением работать.

**Решения:**
- Укажите имя изменения явно: `/opsx:apply add-dark-mode`
- Проверьте, существует ли папка с изменением: `openspec list`
- Убедитесь, что вы находитесь в правильной директории проекта

### «Нет готовых артефактов»

Все артефакты либо уже завершены, либо заблокированы из-за отсутствующих зависимостей.

**Решения:**
- Выполните `openspec status --change <name>`, чтобы увидеть, что вызывает блокировку
- Проверьте, существуют ли требуемые артефакты
- Сначала создайте отсутствующие артефакты-зависимости

### «Схема не найдена»

Указанная схема не существует.

**Решения:**
- Выведите список доступных схем: `openspec schemas`
- Проверьте написание имени схемы
- Создайте схему, если она пользовательская: `openspec schema init <name>`

### Команды не распознаются

ИИ-инструмент не распознаёт команды OpenSpec.

**Решения:**
- Убедитесь, что OpenSpec инициализирован: `openspec init`
- Пересоздайте навыки: `openspec update`
- Проверьте, существует ли директория `.claude/skills/` (для Claude Code)
- Перезапустите ИИ-инструмент, чтобы он подхватил новые навыки

### Артефакты генерируются некорректно

ИИ создаёт неполные или некорректные артефакты.

**Решения:**
- Добавьте контекст проекта в `openspec/config.yaml`
- Добавьте правила для отдельных артефактов для получения конкретных указаний
- Добавьте больше деталей в описание вашего изменения
- Используйте `/opsx:continue` вместо `/opsx:ff` для больше контроля
