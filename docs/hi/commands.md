# कमांंड्स

यह OpenSpec के स्लैश कमांंड्स के लिए संदर्भ है। ये कमांंड्स आपके AI कोडिंग असिस्टेंट के चैट इंटरफेस (जैसे Claude Code, Cursor, Windsurf) में ट्रिगर होती हैं।

वर्कफ़्लो पैटर्न और प्रत्येक कमांंड का उपयोग कब करें, इसके लिए [वर्कफ़्लो](workflows.md) देखें। CLI कमांंड्स के लिए [CLI](cli.md) देखें।

## त्वरित संदर्भ

### डिफ़ॉल्ट क्विक पाथ (`core` प्रोफ़ाइल)

| कमांंड | उद्देश्य |
|---------|---------|
| `/opsx:propose` | एक ही कदम में एक बदलाव बनाएं और प्लानिंग आर्टिफैक्ट्स जनरेट करें |
| `/opsx:explore` | किसी बदलाव को कमिट करने से पहले आइडियाओं पर गहराई से विचार करें |
| `/opsx:apply` | बदलाव के टास्क्स को लागू करें |
| `/opsx:update` | किसी बदलाव की प्लानिंग आर्टिफैक्ट्स को संशोधित करें और उन्हें सुसंगत रखें |
| `/opsx:sync` | डेल्ट स्पेक्स को मुख्य स्पेक्स में मर्ज करें |
| `/opsx:archive` | एक पूर्ण बदलाव को आर्काइव करें |

---

### विस्तारित वर्कफ़्लो कमांंड्स (कस्टम वर्कफ़्लो चयन)

| कमांंड | उद्देश्य |
|---------|---------|
| `/opsx:new` | एक नया बदलाव स्कैफोल्ड शुरू करें |
| `/opsx:continue` | निर्भरताओं के आधार पर अगला आर्टिफैक्ट बनाएं |
| `/opsx:ff` | फास्ट-फ़ॉरवर्ड: एक बार में सभी प्लानिंग आर्टिफैक्ट्स बनाएं |
| `/opsx:verify` | सत्यापन करें कि बदलाव का लागू कार्य आर्टिफैक्ट्स से मेल खाता है |
| `/opsx:bulk-archive` | एक बार में कई बदलावों को आर्काइव करें |
| `/opsx:onboard` | पूरे वर्कफ़्लो के माध्यम से गाइडेड ट्यूटोरियल |

डिफ़ॉल्ट ग्लोबल प्रोफ़ाइल `core` है। विस्तारित वर्कफ़्लो कमांंड्स को सक्षम करने के लिए, `openspec config profile` चलाएं, वर्कफ़्लो चुनें, फिर अपने प्रोजेक्ट में `openspec update` चलाएं।

## कमांड संदर्भ

---

### `/opsx:propose`

एक नया बदलाव बनाएं और एक ही कदम में planning artifacts उत्पन्न करें। यह `core` प्रोफाइल में डिफॉल्ट शुरुआती कमांड है।

**सिंटैक्स:**
```text
/opsx:propose [change-name-or-description]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name-or-description` | नहीं | केबेब-केस नाम या साधारण भाषा में बदलाव का विवरण |

**यह क्या करता है:**
- `openspec/changes/<change-name>/` बनाता है
- कार्यान्वयन से पहले आवश्यक planning artifacts उत्पन्न करता है (`spec-driven` मोड के लिए: proposal, specs, design, tasks)
- तब रुक जाता है जब बदलाव `/opsx:apply` के लिए तैयार हो जाता है

**उदाहरण:**
```text
You: /opsx:propose add-dark-mode

AI:  Created openspec/changes/add-dark-mode/
     ✓ proposal.md
     ✓ specs/ui/spec.md
     ✓ design.md
     ✓ tasks.md
     Ready for implementation. Run /opsx:apply.
```

**सुझाव:**
- सबसे तेज end-to-end पथ के लिए इसका उपयोग करें
- यदि आप कदम-दर-कदम कलाकृति नियंत्रण चाहते हैं, तो विस्तारित वर्कफ़्लो को सक्षम करें और `/opsx:new` + `/opsx:continue` का उपयोग करें

---

### `/opsx:explore`

> **जब आप अनिश्चित हों तो यहीं से शुरुआत करें।** एक्सप्लोर एक बिना जोखिम के सोचने का साथी है: यह आपके कोडबेस को पढ़ता है, विकल्पों की तुलना करता है, और किसी भी बदलाव के मौजूद होने से पहले एक अस्पष्ट विचार को एक ठोस योजना में बदल देता है। यह डिफॉल्ट प्रोफाइल में शामिल है। पूरा केस और अधिक उदाहरणों के लिए [Explore First](explore.md) गाइड देखें।

किसी बदलाव के लिए प्रतिबद्ध होने से पहले विचारों पर चिंतन करें, समस्याओं की जांच करें, और आवश्यकताओं को स्पष्ट करें।

**सिंटैक्स:**
```
/opsx:explore [topic]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `topic` | नहीं | आप जो कुछ एक्सप्लोर या जांचना चाहते हैं |

**यह क्या करता है:**
- बिना किसी संरचना के एक अन्वेषणात्मक संवाद शुरू करता है
- सवालों के जवाब देने के लिए कोडबेस की जांच करता है
- विकल्पों और दृष्टिकोणों की तुलना करता है
- सोच को स्पष्ट करने के लिए दृश्य आरेख बनाता है
- जब अंतर्दृष्टि स्पष्ट हो जाती है तो `/opsx:propose` (डिफॉल्ट) या `/opsx:new` (विस्तारित वर्कफ़्लो) पर स्विच कर सकता है

**उदाहरण:**
```text
You: /opsx:explore

AI:  What would you like to explore?

You: How should we handle authentication for the mobile app?

AI:  Let me investigate your current auth setup...
     [Analyzes codebase]

     I see you're using session-based auth for the web app.
     For mobile, we have three main options:

     1. JWT tokens (stateless, works offline)
     2. OAuth2 (if you need third-party login)
     3. Extend current sessions (simpler, but requires connectivity)

     Your API already has CORS configured. Which direction interests you?

You: Let's go with JWT. Can we start a change for that?

AI:  Ready when you are. Run /opsx:propose add-jwt-auth to begin.
```

**सुझाव:**
- जब आवश्यकताएं अस्पष्ट हों या आपको जांच की आवश्यकता हो तो इसका उपयोग करें
- अन्वेषण के दौरान कोई कलाकृति नहीं बनाई जाती
- निर्णय लेने से पहले कई दृष्टिकोणों की तुलना करने के लिए अच्छा है
- फाइलें पढ़ सकता है और कोडबेस में खोज सकता है

---

### `/opsx:new`

एक नया बदलाव स्कैफोल्ड शुरू करें। बदलाव फोल्डर बनाता है और `/opsx:continue` या `/opsx:ff` से कलाकृतियां उत्पन्न करने के लिए आपका इंतज़ार करता है। यह कमांड विस्तारित वर्कफ़्लो सेट का हिस्सा है (डिफॉल्ट `core` प्रोफाइल में शामिल नहीं है)।

**सिंटैक्स:**
```
/opsx:new [change-name] [--schema <schema-name>]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | बदलाव फोल्डर के लिए नाम (यदि प्रदान नहीं किया गया हो तो पूछा जाता है) |
| `--schema` | नहीं | उपयोग करने के लिए वर्कफ़्लो स्कीमा (डिफॉल्ट: कॉन्फ़िग से या `spec-driven`) |

**यह क्या करता है:**
- `openspec/changes/<change-name>/` निर्देशिका बनाता है
- बदलाव फोल्डर में `.openspec.yaml` मेटाडेटा फाइल बनाता है
- निर्माण के लिए तैयार पहली कलाकृति टेम्पलेट दिखाता है
- यदि प्रदान नहीं किए गए हों तो बदलाव का नाम और स्कीमा पूछता है

**यह क्या बनाता है:**
```
openspec/changes/<change-name>/
└── .openspec.yaml    # बदलाव मेटाडेटा (स्कीमा, बनाए गए तिथि)
```

**उदाहरण:**
```
You: /opsx:new add-dark-mode

AI:  Created openspec/changes/add-dark-mode/
     Schema: spec-driven

     Ready to create: proposal
     Use /opsx:continue to create it, or /opsx:ff to create all artifacts.
```

**सुझाव:**
- वर्णनात्मक नामों का उपयोग करें: `add-feature`, `fix-bug`, `refactor-module`
- `update`, `changes`, `wip` जैसे सामान्य नामों से बचें
- स्कीमा को प्रोजेक्ट कॉन्फ़िग (`openspec/config.yaml`) में भी सेट किया जा सकता है

---

### `/opsx:continue`

निर्भरता श्रृंखला में अगली कलाकृति बनाएं। क्रमिक प्रगति के लिए एक समय में एक कलाकृति बनाता है।

**सिंटैक्स:**
```
/opsx:continue [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को जारी रखना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**
- कलाकृति निर्भरता ग्राफ को क्वेरी करता है
- कौन सी कलाकृतियां तैयार हैं और कौन सी ब्लॉक हैं यह दिखाता है
- पहली तैयार कलाकृति बनाता है
- संदर्भ के लिए निर्भरता फाइलें पढ़ता है
- निर्माण के बाद क्या उपलब्ध होता है यह दिखाता है

**उदाहरण:**
```
You: /opsx:continue

AI:  Change: add-dark-mode

     Artifact status:
     ✓ proposal    (done)
     ◆ specs       (ready)
     ◆ design      (ready)
     ○ tasks       (blocked - needs: specs)

     Creating specs...

     [Reads proposal.md, creates specs based on requirements]

     ✓ Created openspec/changes/add-dark-mode/specs/ui/spec.md

     Now available: tasks
     Run /opsx:continue to create the next artifact.
```

**सुझाव:**
- जब आप आगे बढ़ने से पहले प्रत्येक कलाकृति की समीक्षा करना चाहते हैं तो इसका उपयोग करें
- जब आप नियंत्रण चाहते हैं तो जटिल बदलावों के लिए अच्छा है
- कई कलाकृतियां एक साथ तैयार हो सकती हैं
- जारी रखने से पहले आप बनाई गई कलाकृतियों को संपादित कर सकते हैं

---

### `/opsx:ff`

कलाकृति निर्माण को फास्ट-फॉरवर्ड करें। एक बार में सभी planning artifacts बनाता है।

**सिंटैक्स:**
```
/opsx:ff [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को फास्ट-फॉरवर्ड करना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**
- निर्भरता क्रम में सभी कलाकृतियां बनाता है
- टूडू लिस्ट के माध्यम से प्रगति को ट्रैक करता है
- तब रुक जाता है जब सभी `apply-required` कलाकृतियां पूरी हो जाती हैं
- अगली कलाकृति बनाने से पहले प्रत्येक निर्भरता को पढ़ता है

**उदाहरण:**
```
You: /opsx:ff add-dark-mode

AI:  Fast-forwarding add-dark-mode...

     ✓ Creating proposal.md
     ✓ Creating specs/ui/spec.md
     ✓ Creating design.md
     ✓ Creating tasks.md

     All planning artifacts complete!
     Ready for implementation. Run /opsx:apply to begin.
```

**सुझाव:**
- जब आपको जो बना रहे हैं उसका स्पष्ट चित्र हो तो इसका उपयोग करें
- सरल बदलावों के लिए `/opsx:continue` से तेज है
- बाद में आप अभी भी कलाकृतियों को संपादित कर सकते हैं
- छोटे से मध्यम आकार के फीचर्स के लिए अच्छा है

---

### `/opsx:apply`

बदलाव से कार्यों को लागू करें। कोड लिखते हुए कार्य सूची के माध्यम से काम करता है और आइटमों को चेक करता है।

**सिंटैक्स:**
```
/opsx:apply [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को लागू करना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**
- `tasks.md` को पढ़ता है और अपूर्ण कार्यों की पहचान करता है
- कार्यों को एक-एक करके पूरा करता है
- जरूरत के अनुसार कोड लिखता है, फाइलें बनाता है, परीक्षण चलाता है
- चेकबॉक्स `[x]` के साथ कार्यों को पूर्ण चिह्नित करता है

**उदाहरण:**
```
You: /opsx:apply

AI:  Implementing add-dark-mode...

     Reading tasks.md:
     - [ ] 1.1 Create ThemeContext
     - [ ] 1.2 Add CSS custom properties
     - [ ] 1.3 Implement localStorage persistence

     Working on 1.1: Create ThemeContext...
     [Creates src/context/ThemeContext.tsx]

     ✓ 1.1 Complete

     Working on 1.2: Add CSS custom properties...
     [Updates src/styles/globals.css]

     ✓ 1.2 Complete
     ...
```

**सुझाव:**
- यदि बाधा आ जाए तो आप जहां छोड़े थे वहां से फिर से शुरू कर सकता है
- बदलाव का नाम निर्दिष्ट करके समानांतर बदलावों के लिए उपयोग करें
- पूर्णता की स्थिति `tasks.md` चेकबॉक्स में ट्रैक की जाती है

---

### `/opsx:update`

किसी बदलाव की मौजूदा planning artifacts को संशोधित करें और उन्हें एक दूसरे के साथ सुसंगत रखें। केवल planning artifacts ही - यह कभी भी कोड को संपादित नहीं करता।

**सिंटैक्स:**

```text
/opsx:update [change-name]
```

**आर्ग्यूमेंट:**

| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को अपडेट करना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**

- `openspec status --change <name> --json` के माध्यम से बदलाव की कलाकृतियों को पढ़ता है
- आपके अनुरोधित संशोधन को लागू करता है, या यदि आपने कोई नाम नहीं दिया हो तो विरोधाभासों के लिए कलाकृतियों की समीक्षा करता है
- किसी भी दिशा में अन्य मौजूदा कलाकृतियों को सुसंगत करता है (डिज़ाइन संपादन प्रस्ताव तक पीछे की ओर प्रभाव डाल सकता है)
- लिखने से पहले प्रत्येक संपादन को आपसे पुष्टि करता है, एक समय में एक कलाकृति के लिए
- अगला कदम सुझाकर समाप्त होता है: `/opsx:continue` (कलाकृतियां गुम हैं), `/opsx:apply` (संशोधित योजना को कोड में लागू करें), या `/opsx:archive` (सब कुछ पूरा हो गया)

**उदाहरण:**

```text
You: /opsx:update add-dark-mode - we're storing the theme in a cookie now, not localStorage

AI:  Reading add-dark-mode artifacts...
```

```
/opsx:verify [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |

**यह क्या करता है:**
- कार्यान्वयन गुणवत्ता के तीन आयामों की जांच करता है
- कार्यान्वयन के सबूत के लिए कोडबेस में खोजता है
- समस्याओं को CRITICAL, WARNING या SUGGESTION श्रेणियों में वर्गीकृत करके रिपोर्ट करता है
- आर्काइव को ब्लॉक नहीं करता, लेकिन समस्याओं को सामने लाता है

**सत्यापन आयाम:**

|-----------|-------------------|

| **सुसंगतता** | डिज़ाइन निर्णय कोड में दर्ज हैं, पैटर्न सुसंगत हैं |

- कमिट करने से पहले AI के काम की समीक्षा के लिए अच्छा है

### `/opsx:sync`

**वैकल्पिक कमांड।** किसी बदलाव से डेल्टा स्पेक्स को मुख्य स्पेक्स में मर्ज करें। आर्काइव आवश्यकता पड़ने पर सिंक करने के लिए पूछेगा, इसलिए आपको आमतौर पर इसे मैन्युअली चलाने की आवश्यकता नहीं होती है।

**सिंटैक्स:**
```
/opsx:sync [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को सिंक करना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**
- बदलाव फोल्डर से डेल्टा स्पेक्स को पढ़ता है
- ADDED/MODIFIED/REMOVED/RENAMED सेक्शनों को पार्स करता है
- बदलावों को मुख्य `openspec/specs/` निर्देशिका में मर्ज करता है
- डेल्टा में नहीं उल्लिखित मौजूदा सामग्री को बनाए रखता है
- बदलाव को आर्काइव नहीं करता (सक्रिय रहता है)

**उदाहरण:**
```text
You: /opsx:sync

AI:  Syncing add-dark-mode delta specs...

     Reading: openspec/changes/add-dark-mode/specs/ui/spec.md
     Target:  openspec/specs/ui/spec.md

     Changes to apply:
     ✓ ADDED: Theme Selection requirement (2 scenarios)
     ✓ ADDED: Accessibility requirement (1 scenario)

     Merging...
     ✓ openspec/specs/ui/spec.md updated

     Change remains active. Run /opsx:archive when ready to complete.
```

**मैन्युअली उपयोग करने के परिदृश्य:**

| परिदृश्य | सिंक करें? |
|-----------|-----------|
| लंबे समय तक चलने वाला बदलाव, आर्काइव करने से पहले मुख्य स्पेक्स में स्पेक्स चाहिए | हां |
| कई समानांतर बदलावों को अपडेटेड बेस स्पेक्स की आवश्यकता हो | हां |
| मर्ज को अलग से पूर्वावलोकन/समीक्षा करना चाहते हैं | हां |
| त्वरित बदलाव, सीधे आर्काइव करने जा रहे हैं | नहीं (आर्काइव स्वयं संभालता है) |

**सुझाव:**
- सिंक बुद्धिमान है, कॉपी-पेस्ट नहीं
- बिना डुप्लीकेट किए मौजूदा आवश्यकताओं में परिदृश्य जोड़ सकता है
- सिंक के बाद बदलाव सक्रिय रहता है (आर्काइव नहीं किया जाता है)
- अधिकांश उपयोगकर्ताओं को इसे सीधे कॉल करने की आवश्यकता कभी नहीं होती - आवश्यकता पड़ने पर आर्काइव पूछता है

### `/opsx:archive`

एक पूरे बदलाव को आर्काइव करें। बदलाव को अंतिम रूप देता है और इसे आर्काइव फोल्डर में स्थानांतरित करता है।

**सिंटैक्स:**
```
/opsx:archive [change-name]
```

**आर्ग्यूमेंट:**
| आर्ग्यूमेंट | आवश्यक | विवरण |
|----------|----------|-------------|
| `change-name` | नहीं | किस बदलाव को आर्काइव करना है (यदि प्रदान नहीं किया गया हो तो संदर्भ से निष्कर्षित किया जाता है) |

**यह क्या करता है:**
- कलाकृति पूर्णता की स्थिति की जांच करता है
- कार्य पूर्णता की जांच करता है (अपूर्ण होने पर चेतावनी देता है)
- यदि पहले ही सिंक नहीं किया गया हो तो डेल्टा स्पेक्स सिंक करने का ऑफर करता है
- बदलाव फोल्डर को `openspec/changes/archive/YYYY-MM-DD-<name>/` में स्थानांतरित करता है
- ऑडिट ट्रेल के लिए सभी कलाकृतियों को बनाए रखता है

**उदाहरण:**
```text

AI:  Archiving add-dark-mode...

     Artifact status:
     ✓ proposal.md exists
     ✓ specs/ exists
     ✓ design.md exists
     ✓ tasks.md exists (8/8 tasks complete)

     Delta specs: Not yet synced
     → Sync now? (recommended)

You: Yes

AI:  ✓ Synced specs to openspec/specs/ui/spec.md
     ✓ Moved to openspec/changes/archive/2025-01-24-add-dark-mode/

```

**सुझाव:**
- आर्काइव अपूर्ण कार्यों पर ब्लॉक नहीं करेगा, लेकिन चेतावनी देगा
- डेल्टा स्पेक्स को आर्काइव के दौरान या पहले ही सिंक किया जा सकता है
- आर्काइव किए गए बदलाव इतिहास के लिए बनाए रखे जाते हैं
- समस्याओं को पकड़ने के लिए पहले `/opsx:verify` का उपयोग करें

### `/opsx:onboard`

पूरे OpenSpec वर्कफ़्लो के माध्यम से गाइडेड ऑनबोर्डिंग। आपके वास्तविक कोडबेस का उपयोग करते हुए एक इंटरैक्टिव ट्यूटोरियल।

**सिंटैक्स:**
```
/opsx:onboard
```

**यह क्या करता है:**
- कथन के साथ एक पूरे वर्कफ़्लो चक्र के माध्यम से चलता है
- वास्तविक सुधार के अवसरों के लिए आपके कोडबेस को स्कैन करता है
- वास्तविक कलाकृतियों के साथ एक वास्तविक बदलाव बनाता है
- वास्तविक कार्य लागू करता है (छोटे, सुरक्षित बदलाव)
- पूरे बदलाव को आर्काइव करता है
- जैसे-जैसे कदम होते हैं प्रत्येक कदम की व्याख्या करता है

**चरण:**
1. स्वागत और कोडबेस विश्लेषण
2. सुधार के अवसर की खोज
3. बदलाव बनाना (`/opsx:new`)
4. प्रस्ताव लिखना
5. स्पेक्स बनाना
6. डिज़ाइन लिखना
7. कार्य सूची बनाना
8. कार्य लागू करना (`/opsx:apply`)
9. कार्यान्वयन सत्यापित करना
10. बदलाव आर्काइव करना
11. सारांश और अगले कदम

**उदाहरण:**
```
You: /opsx:onboard

AI:  Welcome to OpenSpec!

     I'll walk you through the complete workflow using your actual codebase.
     We'll find something small to improve, create a proper change for it,
     implement it, and archive it.

     Let me scan your codebase for opportunities...

     [Analyzes codebase]

     I found a few things we could work on:
     1. Add input validation to the contact form
     2. Improve error messages in the auth flow
     3. Add loading states to async buttons

     Which interests you? (or suggest something else)
```

**सुझाव:**
- वर्कफ़्लो सीखने वाले नए उपयोगकर्ताओं के लिए सबसे अच्छा है
- वास्तविक कोड का उपयोग करता है, नाटकीय उदाहरणों का नहीं
- एक वास्तविक बदलाव बनाता है जिसे आप रख सकते हैं या फेंक सकते हैं
- पूरा करने में 15-30 मिनट लगते हैं

## पुराने कमांड्स

ये कमांड्स पुराने "एक बार में सभी" वर्कफ्लो का उपयोग करते हैं। ये अभी भी काम करते हैं लेकिन OPSX कमांड्स की सिफारिश की जाती है।

| कमांड | यह क्या करता है |
|---------|--------------|
| `/openspec:proposal` | एक बार में सभी आर्टिफैक्ट्स बनाएं (प्रस्ताव, स्पेक्स, डिज़ाइन, टास्क) |
| `/openspec:apply` | बदलाव को लागू करें |
| `/openspec:archive` | बदलाव को आर्काइव करें |

**पुराने कमांड्स का उपयोग कब करें:**
- पुराने वर्कफ्लो का उपयोग करने वाले मौजूदा प्रोजेक्ट्स
- आसान बदलाव जहां आपको इंक्रीमेंटल आर्टिफैक्ट निर्माण की जरूरत नहीं है
- सभी या कुछ नहीं दृष्टिकोण की पसंद

**OPSX पर माइग्रेट करना:**
पुराने बदलावों को OPSX कमांड्स के साथ जारी रखा जा सकता है। आर्टिफैक्ट संरचना संगत है।

## समस्या निवारण

### "बदलाव नहीं मिला"

कमांड ने यह पहचानने में असमर्थ रहा कि किस बदलाव पर काम करना है।

**समाधान:**
- बदलाव का नाम स्पष्ट रूप से निर्दिष्ट करें: `/opsx:apply add-dark-mode`
- जांचें कि बदलाव फोल्डर मौजूद है: `openspec list`
- सत्यापित करें कि आप सही प्रोजेक्ट डिरेक्ट्री में हैं

### "कोई आर्टिफैक्ट तैयार नहीं"

सभी आर्टिफैक्ट्स या तो पूर्ण हैं या अनुपस्थित निर्भरताओं द्वारा ब्लॉक हैं।

**समाधान:**
- ब्लॉक करने वाली चीजों को देखने के लिए `openspec status --change <name>` चलाएं
- जांचें कि आवश्यक आर्टिफैक्ट्स मौजूद हैं या नहीं
- पहले अनुपस्थित निर्भरता आर्टिफैक्ट्स बनाएं

### "स्कीमा नहीं मिला"

निर्दिष्ट स्कीमा मौजूद नहीं है।

**समाधान:**
- उपलब्ध स्कीमाओं की सूची बनाएं: `openspec schemas`
- स्कीमा नाम की वर्तनी जांचें
- यदि यह कस्टम है तो स्कीमा बनाएं: `openspec schema init <name>`

### कमांड्स पहचाने नहीं जा रहे

एआई टूल OpenSpec कमांड्स को पहचानता नहीं है।

**समाधान:**
- सुनिश्चित करें कि OpenSpec इनिशियलाइज़ है: `openspec init`
- कौशलों को फिर से जनरेट करें: `openspec update`
- जांचें कि `.claude/skills/` डिरेक्ट्री मौजूद है (Claude Code के लिए)
- नए कौशलों को लेने के लिए अपने एआई टूल को रीस्टार्ट करें

### आर्टिफैक्ट्स सही तरीके से जनरेट नहीं हो रहे हैं

एआई अधूरे या गलत आर्टिफैक्ट्स बनाता है।

**समाधान:**
- `openspec/config.yaml` में प्रोजेक्ट कॉन्टेक्स्ट जोड़ें
- विशिष्ट मार्गदर्शन के लिए प्रत्येक आर्टिफैक्ट के लिए नियम जोड़ें
- अपने बदलाव विवरण में अधिक विवरण प्रदान करें
- अधिक नियंत्रण के लिए `/opsx:ff` के बजाय `/opsx:continue` का उपयोग करें
