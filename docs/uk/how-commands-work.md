# Як працюють команди

**Одне, що треба знати: OpenSpec має два типи команд, і вони виконуються в двох різних місцях.**

- Команди `openspec ...` виконуються у вашому **терміналі**. (Приклад: `openspec init`.)
- Команди `/opsx:...` виконуються у **чаті вашого ШІ-помічника**. (Приклад: `/opsx:propose`.)

Якщо ви коли-небудь введете `/opsx:propose` у свій термінал і нічого не станеться, ця сторінка пояснює чому. Ви спілкуєтеся з неправильною половиною OpenSpec. Команди з косою рискою (slash commands) не є термінальними командами. Це інструкції, які ви даєте своєму ШІ-помічнику з кодування, у тому самому полі чату, де ви зазвичай набираєте "додайте форму входу".

Ця єдина відмінність є найпоширенішою перешкодою для нових користувачів, тому давайте зробимо це кришталево зрозумілим.

## Дві половини

OpenSpec — це один проєкт, що виконує дві ролі.

**CLI (термінальна половина).** Програма на ім'я `openspec`, яку ви встановлюєте та запускаєте з вашої оболонки (shell). Вона налаштовує ваш проєкт, показує список та перевіряє зміни, відображає панель керування (dashboard) та архівує завершену роботу. Ви вводите їх у iTerm, терміналі VS Code, PowerShell, будь-де, де ви запускаєте `git` або `npm`.

```bash
openspec init        # налаштувати OpenSpec у цьому проєкті
openspec list        # переглянути активні зміни
openspec view        # відкрити інтерактивну панель керування
```

**Команди з косою рискою (чатова половина).** Короткі команди, такі як `/opsx:propose` та `/opsx:apply`, які ви вводите у вашому ШІ-помічнику. Вони вказують ШІ дотримуватися робочого процесу OpenSpec: створити пропозицію, написати специфікації, зібрати зі списку завдань, заархівувати після завершення. Ви вводите їх у Claude Code, Cursor, Windsurf, Copilot або будь-якому іншому помічнику, який ви використовуєте.

```text
/opsx:propose add-dark-mode    (введено у чаті вашого ШІ)
/opsx:apply                    (введено у чаті вашого ШІ)
/opsx:archive                  (введено у чаті вашого ШІ)
```

Ось ментальна модель на одному зображенні:

```text
        ВАШ ТЕРМІНАЛ                         ЧАТ ВАШОГО ШІ-ПОМІЧНИКА
   ┌──────────────────────┐               ┌──────────────────────────────┐
   │  $ openspec init     │   встановлює  │  /opsx:propose add-dark-mode  │
   │  $ openspec list     │  ──────────►  │  /opsx:apply                  │
   │  $ openspec view     │   команди     │  /opsx:archive                │
   └──────────────────────┘    і навики   └──────────────────────────────┘
        запускайте openspec тут               запускайте /opsx:* тут
```

Зверніть увагу на стрілку. Запуск `openspec init` у вашому терміналі і *встановлює* команди з косою рискою у ваш ШІ-інструмент. Термінальна половина налаштовує чатову половину. Після цього щоденне керування відбувається переважно у чаті.

## "Як запустити інтерактивний режим?"

**Окремий інтерактивний режим запускати не потрібно.** Це питання виникає часто, тому воно заслуговує на просту відповідь.

Ви не входите в спеціальний режим OpenSpec. Ви просто відкриваєте свого ШІ-помічника з кодування, як зазвичай, і вводите команду з косою рискою в чат. Команда з косою рискою *і є* тим, як ви "заходите" в OpenSpec. Ваш помічник розпізнає її, завантажує відповідний навик (skill) OpenSpec і починає слідувати робочому процесу.

Тож справжні інструкції такі:

1. Відкрийте вашого ШІ-помічника з кодування (Claude Code, Cursor, Windsurf тощо) у вашому проєкті.
2. Введіть `/opsx:propose` у його чат, у те саме місце, де ви вводите будь-який інший запит.
3. Погляньте на автодоповнення: якщо OpenSpec встановлено, ви побачите `/opsx:propose`, `/opsx:apply` та інші команди, що з'являтимуться під час введення косої риски.

От і все. Немає режиму для перемикання, демона для запуску, окремого вікна.

Одна річ, яка *дійсно* є інтерактивною, знаходиться в терміналі: `openspec view`. Вона відкриває панель керування для перегляду ваших специфікацій та змін. Але це лише переглядач, а не те, за допомогою чого ви пропонуєте та будуєте. Будівництво відбувається через команди з косою рискою в чаті.

## Чому цей поділ існує

Це варто зрозуміти, оскільки це пояснює, чому OpenSpec працює з більш ніж 25 різними ШІ-інструментами.

CLI — це **двигун**. Він знає правила: як виглядає папка змін, які артефакти залежать від чого, як об'єднати дельта-специфікацію у ваше джерело істини. Він однаковий скрізь.

Команди з косою рискою — це **кермо**, і кожен ШІ-інструмент має трохи інше. Claude Code називає їх командами. Cursor та Windsurf мають власні формати. Деякі інструменти називають їх навиками (skills). Коли ви запускаєте `openspec init`, OpenSpec генерує правильний тип файлу для кожного інструменту, який ви обрали, тому однаковий намір `/opsx:propose` працює незалежно від того, якому помічнику ви віддаєте перевагу.

Сильна сторона цього дизайну: ви вивчаєте робочий процес один раз і використовуєте його в різних інструментах. Компроміс: точний синтаксис команди може трохи відрізнятися між інструментами, про що йдеться в наступному розділі.

## Синтаксис команд з косою рискою для різних інструментів

Намір однаковий скрізь. Пунктуація відрізняється. Використовуйте форму, що відповідає вашому помічнику.

| Інструмент | Як ви їх вводите |
|------|-----------------|
| Claude Code | `/opsx:propose`, `/opsx:apply` |
| Cursor | `/opsx-propose`, `/opsx-apply` |
| Windsurf | `/opsx-propose`, `/opsx-apply` |
| GitHub Copilot (IDE) | `/opsx-propose`, `/opsx-apply` |
| CodeArts | стиль навиків, напр. `/openspec-propose` |
| Codex | стиль навиків через `.codex/skills/openspec-*` |
| Oh My Pi | `/opsx-propose`, `/opsx-apply` |
| Kimi CLI | стиль навиків, напр. `/skill:openspec-propose` |
| Trae | `/opsx-propose`, `/opsx-apply` |

Більшість інструментів використовують або форму з двокрапкою (`/opsx:propose`), або форму з дефісом (`/opsx-propose`). Деякі інструменти показують OpenSpec як іменовані навики замість команд з косою рискою; для них ви викликаєте навик за назвою. Повний список для кожного інструменту, включаючи точне розташування файлів, що записуються, знаходиться у розділі [Підтримувані інструменти](supported-tools.md).

Якщо сумніваєтеся, введіть косу риску у чаті вашого ШІ та подивіться на автодоповнення. Ваш інструмент покаже вам форму, яку він очікує.

## Як ті команди туди потрапили: навики та команди

Коли ви запускаєте `openspec init` (або `openspec update`), OpenSpec записує невеликі файли у ваш проєкт, щоб ваш ШІ-інструмент міг знайти робочий процес. Залежно від вашого інструменту та налаштувань, це **навики (skills)**, **команди** або й те, й інше.

- **Навики (skills)** знаходяться в таких місцях, як `.claude/skills/openspec-*/SKILL.md`. Це новий міжінструментальний стандарт: папка з інструкціями, які ваш помічник автоматично виявляє.
- **Команди** знаходяться в таких місцях, як `.claude/commands/opsx/<id>.md`. Це старіші файли команд з косою рискою для кожного інструменту. Codex не генерує файлів команд; використовуйте `.codex/skills/openspec-*`.

Вам не потрібно перейматися тим, який із них використовує ваш інструмент. Ви просто вводите команду з косою рискою, і вона працює. Але знання про існування цих файлів допомагає, коли щось йде не так: якщо ваші команди зникають, це зазвичай означає, що ці файли відсутні або застарілі, і `openspec update` відновлює їх.

Дивіться [Підтримувані інструменти](supported-tools.md) для точних шляхів для кожного інструменту та [Посібник з міграції](migration-guide.md), щоб дізнатися, як навики замінили старіший підхід лише з командами.

## Підтвердження встановлення

Швидкі перевірки, від найшвидшої:

1. **Введіть косу риску у чаті вашого ШІ.** Почніть вводити `/opsx` і спостерігайте за пропозиціями автодоповнення. Якщо вони з'являються, все готово.
2. **Перевірте файли.** Для Claude Code переконайтеся, що `.claude/skills/` містить папки `openspec-*`. Інші інструменти використовують власні каталоги ([Підтримувані інструменти](supported-tools.md) містять їх список).
3. **Запустіть налаштування знову.** З кореня вашого проєкту запустіть `openspec update`. Це відновить файли навиків та команд для всіх налаштованих вами інструментів.
4. **Перезапустіть вашого помічника.** Багато інструментів сканують навики та команди під час запуску, тому свіже вікно може бути відсутнім кроком.

## Які взагалі команди в мене є?

За замовчуванням OpenSpec встановлює **основний** набір команд з косою рискою:

- `/opsx:explore`: продумати ідею з ШІ перед тим, як зробити зміну (чудовий перший крок, коли ви не впевнені)
- `/opsx:propose`: створити зміну та скласти всі її артефакти планування за один крок
- `/opsx:apply`: побудувати зміну, працюючи зі списком її завдань
- `/opsx:sync`: об'єднати оновлення специфікацій зміни з вашими основними специфікаціями (зазвичай автоматично)
- `/opsx:archive`: завершити зміну та заархівувати її

Гарний ритм за замовчуванням: `explore`, коли ви з'ясовуєте, що робити, потім `propose`, `apply`, `archive`. Посібник [Спочатку дослідіть](explore.md) пояснює, чому цей початковий крок виправдає себе.

Також є **розширений** набір для тих, хто хоче більш детального контролю (`/opsx:new`, `/opsx:continue`, `/opsx:ff`, `/opsx:verify`, `/opsx:bulk-archive`, `/opsx:onboard`). Ви вмикаєте його за допомогою `openspec config profile`, а потім застосовуєте за допомогою `openspec update`.

Новачок у всьому цьому? `/opsx:onboard` (у розширеному наборі) проведе вас через повну зміну у вашій власній кодовій базі, коментуючи кожен крок. Це найдружніше можливе знайомство.

Щодо того, що робить кожна команда детальніше, дивіться [Команди](commands.md). Щодо того, коли яку використовувати, дивіться [Робочі процеси](workflows.md).

## Чистий перший запуск

Збираючи все разом, ось вся послідовність з кожним кроком, позначеним залежно від того, де він відбувається.

```text
ТЕРМІНАЛ   $ npm install -g @fission-ai/openspec@latest
ТЕРМІНАЛ   $ cd your-project
ТЕРМІНАЛ   $ openspec init
              (встановлює команди з косою рискою у ваш ШІ-інструмент)

ЧАТ ШІ      /opsx:explore
              (опціонально: спочатку продумати ідею з ШІ)

ЧАТ ШІ      /opsx:propose add-dark-mode
              (ШІ складає пропозицію, специфікації, дизайн, завдання)

ЧАТ ШІ      /opsx:apply
              (ШІ будує це, відмічаючи завдання)

ЧАТ ШІ      /opsx:archive
              (зміна об'єднується з вашими специфікаціями та архівується)
```

Два кроки в терміналі для налаштування. Потім ви живете в чаті. Такий ритм.
