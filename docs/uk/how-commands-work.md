# Як працюють команди

**Є одна річ, яку потрібно знати: OpenSpec має два типи команд, і вони виконуються в двох різних місцях.**

- Команди `openspec ...` виконуються у вашому **терміналі**. (Приклад: `openspec init`.)
- Команди `/opsx:...` виконуються у **чаті вашого AI-асистента**. (Приклад: `/opsx:propose`.)

Якщо ви коли-небудь вводите `/opsx:propose` у свій термінал, і нічого не відбувається, ця сторінка пояснює чому. Ви спілкуєтеся з неправильною частиною OpenSpec. Склаш-команди (slash commands) — це не команди терміналу. Це інструкції, які ви даєте своєму AI-асистенту для кодування у тому ж чаті, де зазвичай вводите "додати форму входу".

Ця одна відмінність є найбільшою перешкодою для нових користувачів, тому давайте прояснимо це максимально чітко.

## Дві частини

OpenSpec — це один проєкт у двох ролях.

**CLI (термінальна частина).** Це програма з назвою `openspec`, яку ви встановлюєте та запускаєте зі своєї оболонки (shell). Вона налаштовує ваш проєкт, перераховує та валідує зміни, відображає дашборд і архівує завершену роботу. Ви вводите це у iTerm, термінал VS Code, PowerShell — будь-де, де ви б запускали `git` або `npm`.

```bash
openspec init        # налаштовує OpenSpec у цьому проєкті
openspec list        # переглядає активні зміни
openspec view        # відкриває інтерактивний дашборд
```

**Склаш-команди (чаткова частина).** Це короткі команди, як `/opsx:propose` та `/opsx:apply`, які ви вводите у свого AI-асистента. Вони наказують ШІ слідувати робочому процесу OpenSpec: створити пропозицію, написати специфікації, побудувати на основі списку завдань, заархівувати після завершення. Ви вводите їх у Claude Code, Cursor, Windsurf, Copilot або будь-якого іншого асистента.

```text
/opsx:propose add-dark-mode    (введено у чаті AI)
/opsx:apply                    (введено у чаті AI)
/opsx:archive                  (введено у чаті AI)
```

Ось ментальна модель на одній картинці:

```text
        ВАШ ТЕРМІНАЛ                         ЧАТ ВАШОГО AI-АСИСТЕНТА
   ┌──────────────────────┐               ┌──────────────────────────────┐
   │  $ openspec init     │   інсталює    │  /opsx:propose add-dark-mode  │
   │  $ openspec list     │  ──────────►  │  /opsx:apply                  │
   │  $ openspec view     │   команди      │  /opsx:archive                │
   └──────────────────────┘    & навички   └──────────────────────────────┘
        запускає openspec тут                       запускає /opsx:* тут
```

Зверніть увагу на стрілку. Запуск `openspec init` у вашому терміналі *інсталює* склаш-команди у ваш інстру AI. Термінальна частина налаштовує чаткову частину. Після цього щоденна робота переважно відбувається в чаті.

## "Як мені розпочати інтерактивний режим?"

**Окремого інтерактивного режиму немає.** Це питання виникає часто, тому воно заслуговує на просту відповідь.

Ви не заходите в спеціальний режим OpenSpec. Ви просто відкриваєте свого AI-кодувального асистента, як завжди, і вводите склаш-команду у чат. Склаш-команда *є* тим способом, яким ви "заходите" в OpenSpec. Ваш асистент розпізнає її, завантажує відповідну навичку (skill) OpenSpec та починає слідувати робочому процесу.

Тож справжні інструкції такі:

1. Відкрийте свого AI-кодувального асистента (Claude Code, Cursor, Windsurf тощо) у вашому проєкті.
2. Введіть `/opsx:propose` у його чат, там же, де ви вводите будь-який інший запит.
3. Спостерігайте за автозаповненням: якщо OpenSpec встановлено, ви побачите `/opsx:propose`, `/opsx:apply` та інших, коли будете вводити склаш.

Ось і все. Немає режиму для перемикання, не потрібно запускати демона, немає окремого вікна.

Одна річ, яка справді інтерактивна — це `openspec view`. Вона відкриває дашборд для перегляду ваших специфікацій та змін. Але це лише переглядач, а не те, чим ви пропонуєте і будуєте. Будівництво відбувається через склаш-команди в чаті.

## Чому існує цей поділ

Це варто зрозуміти, оскільки це пояснює, чому OpenSpec працює з 25+ різними інструментами ШІ.

CLI — це **двигун**. Він знає правила: як виглядає папка зі змінами, які артефакти залежать від яких, як об'єднати дельта-специфікацію у ваш джерело істини. Це однакове скрізь.

Склаш-команди — це **кермо**, і кожен інструмент ШІ має своє трохи відмінне кермо. Claude Code називає їх командами. Cursor та Windsurf мають власні формати. Деякі інструменти називають їх навичками (skills). Коли ви запускаєте `openspec init`, OpenSpec генерує правильний тип файлів для кожного обраного вами інструменту, тому одна й та сама інтенція `/opsx:propose` працює незалежно від того, якого асистента ви віддаєте перевагу.

Сила цього дизайну полягає в тому, що ви вивчаєте робочий процес лише один раз і застосовуєте його до всіх інструментів. Компромісник: точна синтаксика команди може трохи відрізнятися між інструментами, про що йде наступний розділ.

## Синтаксис склаш-команд за інструментами

Інтенція (мета) однакова скрізь. Пунктуація відрізняється. Використовуйте форму, яка відповідає вашому асистенту.

| Tool | Як ви це вводите |
|------|-----------------|
| Claude Code | `/opsx:propose`, `/opsx:apply` |
| Cursor | `/opsx-propose`, `/opsx-apply` |
| Windsurf | `/opsx-propose`, `/opsx-apply` |
| GitHub Copilot (IDE) | `/opsx-propose`, `/opsx-apply` |
| Kimi CLI | skill-style, наприклад, `/skill:openspec-propose` |
| Trae | skill-style, наприклад, `/openspec-propose` |

Більшість інструментів використовують або формат із двокрапкою (`/opsx:propose`), або формат із дефісом (`/opsx-propose`). Кілька інструментів відображають OpenSpec як названі навички, а не склаш-команди; для них ви викликаєте навичку за назвою. Повний перелік за інструментами, включаючи точно які файли створюються і де, міститься у [Supported Tools](supported-tools.md).

Якщо сумніваєтеся, введіть склаш у свій чат AI та подивіться на автозаповнення. Ваш інструмент покаже вам очікуваний формат.

## Як команди з'явилися: навички та команди

Коли ви запускаєте `openspec init` (або `openspec update`), OpenSpec створює невеликі файли у вашому проєкті, щоб ваш інструмент ШІ міг знайти робочий процес. Залежно від вашого інструменту та налаштувань це є **навичками** (skills), **командами** (commands) або обома.

- **Навички** живуть у таких місцях, як `.claude/skills/openspec-*/SKILL.md`. Це виникаючий кросплатформний стандарт: папка з інструкціями, які автоматично виявляє ваш асистент.
- **Команди** живуть у таких місцях, як `.claude/commands/opsx/<id>.md`. Це старіші файли склаш-команд для кожного інструменту.

Вам не потрібно турбуватися, який з них використовує ваш інструмент. Ви просто вводите склаш-команду, і вона працює. Але знання про існування цих файлів допомагає, коли щось йде не так: якщо ваші команди зникають, це зазвичай означає, що ці файли відсутні або застарілі, і `openspec update` їх регенерує.

Дивіться [Supported Tools](supported-tools.md) для точних шляхів за інструментами та [Migration Guide](migration-guide.md), як навички замінили старий підхід лише з командами.

## Підтвердження встановлення

Швидкі перевірки, спочатку найпростіші:

1. **Введіть склаш у свій чат AI.** Почніть вводити `/opsx` та стежте за пропозиціями автозаповнення. Якщо вони з'являються, ви готові.
2. **Шукайте файли.** Для Claude Code перевірте, чи містить `.claude/skills/` папки `openspec-*`. Інші інструменти використовують власні директорії ([Supported Tools](supported-tools.md) їх перелічує).
3. **Повторно запустіть налаштування.** З кореня вашого проєкту виконайте `openspec update`. Це регенерує файли навичок та команд для тих інструментів, які ви налаштували.
4. **Перезапустіть свого асистента.** Багато інструментів сканують навички та команди при запуску, тому свіже вікно може бути відсутнім кроком.

## Які команди мені доступні?

За замовчуванням OpenSpec встановлює **основний** набір склаш-команд:

- `/opsx:explore`: обдумайте ідею з ШІ, перш ніж робити зміни (відмінний перший крок, коли ви не впевнені)
- `/opsx:propose`: створіть зміну та підготуйте всі її планувальні артефакти за один раз
- `/opsx:apply`: побудуйте зміну, проходячи через список завдань
- `/opsx:sync`: об'єднайте оновлення специфікації зміни у ваші основні специфікації (зазвичай це автоматично)
- `/opsx:archive`: завершіть зміну та заархівуйте її

Хороший стандартний ритм: `explore`, коли ви розбираєте, що робити, потім `propose`, `apply`, `archive`. Гайд [Explore First](explore.md) пояснює, чому цей стартовий крок того вартий.

Також існує **розширений** набір для тих, хто хоче більшого контролю (`/opsx:new`, `/opsx:continue`, `/opsx:ff`, `/opsx:verify`, `/opsx:bulk-archive`, `/opsx:onboard`). Ви вмикаєте його за допомогою `openspec config profile`, а потім застосовуєте з `openspec update`.

Новачок у цьому? `/opsx:onboard` (у розширеному наборі) проведе вас через повну зміну у вашій кодовій базі, описуючи кожен крок. Це найбільш дружній можливий вступ.

Для детального опису того, що робить кожна команда, дивіться [Commands](commands.md). Для визначення, коли використовувати яку, дивіться [Workflows](workflows.md).

## Чистий перший запуск

Підсумовуючи, ось повна послідовність із позначкою місця, де відбувається кожен крок.

```text
TERMINAL   $ npm install -g @fission-ai/openspec@latest
TERMINAL   $ cd your-project
TERMINAL   $ openspec init
              (інсталює склаш-команди у ваш інстру AI)

AI CHAT      /opsx:explore
              (необов'язково: обміркуйте ідею з ШІ спочатку)

AI CHAT      /opsx:propose add-dark-mode
              (ШІ готує пропозицію, специфікації, дизайн, завдання)

AI CHAT      /opsx:apply
              (ШІ будує це, відмічаючи завдання)

AI CHAT      /opsx:archive
              (зміна об'єднана у ваші специфікації та заархівована)
```

Два термінальні кроки для налаштування. Потім ви працюєте в чаті. Це ритм роботи.
