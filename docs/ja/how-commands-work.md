# コマンドの仕組み

**知っておくべき唯一のポイント: OpenSpecには2種類のコマンドがあり、それぞれ異なる場所で実行されます。**

- `openspec ...` コマンドは**ターミナル**で実行されます。（例: `openspec init`）
- `/opsx:...` コマンドは**AIアシスタントのチャット**で実行されます。（例: `/opsx:propose`）

もしターミナルに `/opsx:propose` と入力しても何も起こらない場合、このページがその理由を説明しています。あなたはOpenSpecの誤った側面と対話しています。スラッシュコマンドはターミナルコマンドではありません。それらは「ログインフォームを追加して」と通常チャットに入力するのと同じように、AIコーディングアシスタントに与える指示です。

この単一の区別は新しいユーザーにとって最も一般的なつまずきの原因なので、明確に説明しましょう。

## 2つの構成部分

OpenSpecは1つのプロジェクトが2つの顔を持っています。

**CLI（ターミナル側）。** `openspec` という名前のプログラムで、シェルからインストールして実行します。プロジェクトのセットアップ、変更の一覧表示と検証、ダッシュボードの表示、完了した作業のアーカイブを行います。これらは `git` や `npm` を実行するのと同じように、iTerm、VS Codeターミナル、PowerShellなどに入力します。

```bash
openspec init        # このプロジェクトにOpenSpecをセットアップ
openspec list        # アクティブな変更を表示
openspec view        # インタラクティブダッシュボードを開く
```

**スラッシュコマンド（チャット側）。** `/opsx:propose` や `/opsx:apply` のような短いコマンドで、AIアシスタントに入力します。これらはAIにOpenSpecワークフローに従うよう指示します: 提案書の作成、仕様書の作成、タスクリストからのビルド、完了時のアーカイブ。これらはClaude Code、Cursor、Windsurf、Copilot、または使用しているアシスタントのチャットに入力します。

```text
/opsx:propose add-dark-mode    （AIチャットに入力）
/opsx:apply                    （AIチャットに入力）
/opsx:archive                  （AIチャットに入力）
```

これが1つの図で表したメンタルモデルです:

```text
        あなたのターミナル                     AIアシスタントのチャット
   ┌──────────────────────┐               ┌──────────────────────────────┐
   │  $ openspec init     │   インストール   │  /opsx:propose add-dark-mode  │
   │  $ openspec list     │  ──────────►  │  /opsx:apply                  │
   │  $ openspec view     │   コマンドと    │  /opsx:archive                │
   └──────────────────────└   スキル       └──────────────────────────────┘
       ここでopenspecを実行                    ここで/opsx:*を実行
```

矢印に注目してください。ターミナルで `openspec init` を実行すると、AIツールにスラッシュコマンドが*インストール*されます。ターミナル側がチャット側をセットアップします。その後、日常的な操作のほとんどはチャットで行われます。

## 「インタラクティブモードを起動するには？」

**起動するための別のインタラクティブモードは存在しません。** この質問は非常に多く寄せられるので、明確な答えを記載します。

特別なOpenSpecモードに入る必要はありません。いつも通りAIコーディングアシスタントを開き、チャットにスラッシュコマンドを入力するだけです。スラッシュコマンドこそがOpenSpecに「入る」方法です。あなたのアシスタントはそれを認識し、対応するOpenSpecスキルを読み込み、ワークフローに従い始めます。

したがって、実際の手順は以下の通りです:

1. AIコーディングアシスタント（Claude Code、Cursor、Windsurfなど）をプロジェクトで開きます。
2. チャットに `/opsx:propose` と入力します。これは他のリクエストを入力するのと同じ場所です。
3. オートコンプリートを確認してください: OpenSpecがインストールされている場合、スラッシュを入力すると `/opsx:propose`、`/opsx:apply` などが表示されます。

以上です。切り替えるモードも、起動するデーモンも、別のウィンドウも必要ありません。

本当にインタラクティブな機能は1つだけターミナルに存在します: `openspec view` です。これは仕様書や変更を閲覧するためのダッシュボードを開きます。しかしこれはビューアーであり、提案やビルドを行うものではありません。ビルドはチャットのスラッシュコマンドを通じて行われます。

## この分割が存在する理由

これは理解する価値があります。なぜなら、OpenSpecが25種類以上の異なるAIツールで動作する理由を説明しているからです。

CLIは**エンジン**です。ルールを知っています: 変更フォルダの構造、アーティファクト間の依存関係、差分仕様を信頼できる情報源にマージする方法など。どこでも同じです。

スラッシュコマンドは**ハンドル**であり、すべてのAIツールには少しずつ異なるハンドルがあります。Claude Codeはコマンドと呼び、CursorとWindsurfは独自の形式を持ち、一部のツールはスキルと呼びます。`openspec init` を実行すると、OpenSpecは選択した各ツール用に適切な種類のファイルを生成するため、同じ `/opsx:propose` の意図が使用するアシスタントに関わらず機能します。

この設計の強み: ワークフローを1回学べば、ツール間で持ち運べます。トレードオフ: コマンドの正確な構文はツールによって若干異なる場合があり、それが次のセクションです。

## ツール別スラッシュコマンド構文

意図はどこでも同じです。句読点が異なります。使用しているアシスタントに一致する形式を使用してください。

| ツール | 入力方法 |
|------|---------|
| Claude Code | `/opsx:propose`, `/opsx:apply` |
| Cursor | `/opsx-propose`, `/opsx-apply` |
| Windsurf | `/opsx-propose`, `/opsx-apply` |
| GitHub Copilot (IDE) | `/opsx-propose`, `/opsx-apply` |
| CodeArts | スキル形式、例: `/openspec-propose` |
| Codex | `.codex/skills/openspec-*` を介したスキル形式 |
| Oh My Pi | `/opsx-propose`, `/opsx:apply` |
| Kimi CLI | スキル形式、例: `/skill:openspec-propose` |
| Trae | `/opsx-propose`, `/opsx-apply` |

ほとんどのツールはコロン形式（`/opsx:propose`）またはダッシュ形式（`/opsx-propose`）のいずれかを使用します。一部のツールはOpenSpecをスラッシュコマンドではなく名前付きスキルとして表示します。その場合はスキル名で呼び出します。正確にどのファイルがどこに書き込まれるかを含むツール別の完全なリストは、[サポートツール](supported-tools.md)にあります。

わからない場合は、AIチャットにスラッシュを入力してオートコンプリートを確認してください。ツールが期待する形式が表示されます。

## コマンドが配置される仕組み: スキルとコマンド

`openspec init`（または `openspec update`）を実行すると、OpenSpecはAIツールがワークフローを見つけられるようにプロジェクトに小さなファイルを書き込みます。ツールと設定によって、これらは**スキル**、**コマンド**、またはその両方です。

- **スキル**は `.claude/skills/openspec-*/SKILL.md` などの場所に存在します。これらは登場しつつあるクロスツール標準です: アシスタントが自動検出する命令のフォルダです。
- **コマンド**は `.claude/commands/opsx/<id>.md` などの場所に存在します。これらは古いツール別スラッシュコマンドファイルです。Codexにはコマンドファイルが生成されません。`.codex/skills/openspec-*` を使用してください。

ツールがどちらを使用するかを気にする必要はありません。スラッシュコマンドを入力するだけで機能します。しかし、これらのファイルが存在することを知っておくと、問題が発生したときに役立ちます: コマンドが消えた場合、通常はこれらのファイルが欠落しているか古いことが原因であり、`openspec update` で再生成されます。

ツール別の正確なパスは[サポートツール](supported-tools.md)に、スキルが古いコマンドのみのアプローチをどのように置き換えたかは[移行ガイド](migration-guide.md)に記載されています。

## インストールされていることの確認

最速の順に簡単な確認方法:

1. **AIチャットにスラッシュを入力してください。** `/opsx` と入力し始め、オートコンプリート候補を確認してください。表示されれば準備完了です。
2. **ファイルを確認してください。** Claude Codeの場合、`.claude/skills/` に `openspec-*` フォルダが含まれていることを確認してください。他のツールは独自のディレクトリを使用します（[サポートツール](supported-tools.md)に一覧があります）。
3. **セットアップを再実行してください。** プロジェクトルートから `openspec update` を実行します。これにより、設定したツール用にスキルとコマンドファイルが再生成されます。
4. **アシスタントを再起動してください。** 多くのツールは起動時にスキルとコマンドをスキャンするため、新しいウィンドウが欠落しているステップである可能性があります。

## デフォルトでどのコマンドが利用可能か？

デフォルトで、OpenSpecは**コア**セットのスラッシュコマンドをインストールします:

- `/opsx:explore`: 変更にコミットする前にAIとアイデアを検討する（確信が持てない場合の最初のステップとして最適）
- `/opsx:propose`: 変更を作成し、すべての計画アーティファクトを1ステップでドラフトする
- `/opsx:apply`: タスクリストを順番に処理して変更をビルドする
- `/opsx:sync`: 変更の仕様更新をメインの仕様にマージする（通常は自動）
- `/opsx:archive`: 変更を完了してファイルに整理する

優れたデフォルトのリズム: 何をするか考えているときは `explore`、次に `propose`、`apply`、`archive`。[まず探索](explore.md)ガイドでは、この最初のステップがなぜ有益かを説明しています。

より細かい制御を希望する人のための**拡張**セットもあります（`/opsx:new`、`/opsx:continue`、`/opsx:ff`、`/opsx:verify`、`/opsx:bulk-archive`、`/opsx:onboard`）。`openspec config profile` で有効にし、`openspec update` で適用します。

初めての方は、拡張セットの `/opsx:onboard` が独自のコードベースで完全な変更を案内し、各ステップを解説します。これは最も親切な導入方法です。

各コマンドの詳細な動作については[コマンド](commands.md)を、いつどれを使うかについては[ワークフロー](workflows.md)を参照してください。

## 最初の実行フロー

まとめると、各ステップが発生する場所を示す完全なシーケンスは以下の通りです。

```text
ターミナル   $ npm install -g @fission-ai/openspec@latest
ターミナル   $ cd your-project
ターミナル   $ openspec init
              （AIツールにスラッシュコマンドをインストール）

AIチャット      /opsx:explore
              （オプション: 最初にAIとアイデアを検討）

AIチャット      /opsx:propose add-dark-mode
              （AIが提案書、仕様書、設計、タスクを作成）

AIチャット      /opsx:apply
              （AIがビルドし、タスクをチェックオフ）

AIチャット      /opsx:archive
              （変更が仕様にマージされ、ファイルに整理される）
```

セットアップに必要なターミナルステップは2つだけです。その後はチャットで作業を続けます。これがリズムです。
